# Tiny Second-hand Shopping Platform
23반-석은호

처음 코드를 읽어보니 비밀번호가 평문으로 저장되고 CSRF 보호가 없는 등 취약한 부분이 다소 발견되었다.
그 부분들을 하나씩 고쳐가고 기능 또한 실제 중고거래 서비스를 참고해 유사한 수준으로 웹서비스를 만들었다.

# 1. 실행 방법

## 1.1. 환경 설정

Ubuntu 기준 Python은 3.9 이상, conda와 venv 중 편한 쪽을 쓰면된다.

conda를 쓴다면:

```bash
git clone https://github.com/eunho87/secure-coding
cd secure-coding
conda env create -f enviroments.yaml
conda activate secure_coding
```

venv를 쓴다면:

```bash
git clone https://github.com/eunho87/secure-coding
cd secure-coding
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## 1.2. 실행하기

```bash
python app.py
```

실행 됐다면 http://127.0.0.1:5000 으로 접속할 수 있다. `market.db`는 처음 실행할 때 자동으로 만들어진다.

등록된 상품은 누구나 볼 수 있어야 한다는 규정에 맞춰 상품 목록과 상세, 검색은 로그인하지 않아도 볼 수 있도록 하였다.
대신 구매나 채팅, 찜하기와 같은 것은 로그인을 해야하도록 구현하였다.

## 1.3. 데모 데이터 넣기

빈 화면에서 시작하면 사고파는 과정을 보여주기가 번거롭기에 상품과 계정을 미리 채워주는 스크립트를 만들어 두었다.
서버를 한 번 실행해 DB가 생긴 뒤에 다른 터미널에서 실행해야 한다.

```bash
python seed.py            # 상품 15개 + 계정 4개 + 사진 등록
python seed.py --reset    # 기존 데모 데이터를 지우고 다시 등록
```

상품 사진은 카테고리에 맞는 무료 이미지를 받아오도록 하였다.

계정은 네 개가 만들어지고 비밀번호는 모두 `Passw0rd1` 이다.

| 계정 | 역할 |
|---|---|
| `daangn_kim`, `market_lee`, `hanguk_park` | 판매자 |
| `buyer_choi` | 구매자 |

## 1.4. 관리자 계정

관리자는 첫 실행 때 자동으로 생성된다. 비밀번호는 `ADMIN_PASSWORD` 환경변수 값을 사용하며
지정하지 않으면 `Admin123!`로 생성된다. 다만 이 기본 비밀번호로 생성된 경우에는
첫 로그인에서 비밀번호를 반드시 바꿔야 다른 페이지로 넘어갈 수 있게 막아두었다.

```bash
ADMIN_PASSWORD='생성할 admin 비밀번호' python app.py
```

## 1.5. 테스트 돌리기

기능과 보안이 실제로 동작하는지 확인하는 스크립트를 `tests/` 아래에 네 개 두었다.
전부 bash와 curl로만 되어 있어서 별도 테스트 프레임워크나 추가 패키지 없이, 실행 중인 서버에
실제 HTTP 요청을 보내고 필요하면 `market.db`를 직접 조회해 결과를 확인한다.

| 파일 | 무엇을 확인하는가 | 케이스 |
|---|---|---|
| `tests/e2e_test.sh` | 가입과 로그인부터 상품 등록, 비로그인 열람, 검색, 찜, 가격 제안, 에스크로 거래, 후기, 신고 누적 자동 차단과 휴면, 관리자 처리와 분쟁 중재, PG 충전까지 전체 흐름. 여기에 CSRF와 SQL 인젝션, XSS, IDOR 같은 공격 시도를 섞어두었다 | 81 |
| `tests/extra_test.sh` | 미인증 소켓 연결 거부, 파일 업로드 검증(텍스트를 이미지로 위장한 파일 거부), 2단계 인증 전체 흐름 | 8 |
| `tests/chat_test.sh` | 채팅. Socket.IO 프로토콜을 직접 주고받아 전체 채팅 브로드캐스트, 1:1 전달과 방 격리(제3자 비노출), 새로고침 후 내역 보존, 길이 제한을 확인한다 | 10 |
| `tests/finance_test.sh` | 동시성과 금융 무결성. 같은 거래에 요청을 20개 동시에 던져 이중 지급이나 이중 충전이 없는지, 원장 합계와 실제 잔액이 맞는지를 DB에서 직접 조회해 확인한다 | 6 |

서버를 띄운 뒤 순서대로 실행하면 된다. 로그인 시도 횟수 제한(IP당 5분에 10회)이 자동화 테스트와
충돌하기 때문에, 테스트용 서버는 `IP_LOGIN_LIMIT`을 올려서 띄워야 한다.

```bash
IP_LOGIN_LIMIT=1000 python app.py &
bash tests/e2e_test.sh
bash tests/extra_test.sh
bash tests/chat_test.sh
bash tests/finance_test.sh
```

각 스크립트는 케이스마다 `PASS` 또는 `FAIL`을 출력하고 마지막에 합계를 찍는다.
`IP_LOGIN_LIMIT`을 빼먹고 돌리면 중간부터 로그인이 막혀 다수가 FAIL로 표시되니 주의해야 한다.
`finance_test.sh`는 `~/secure-coding/market.db`를 직접 읽도록 되어 있어서, 저장소를 다른
경로에 두었다면 스크립트 상단의 `DB` 변수를 고쳐야 한다.

케이스별 상세 목록과 최종 결과는 [2.4 체크리스트 작성 및 테스팅](#24-체크리스트-작성-및-테스팅)에 정리했다.

# 2. 개발 전 과정 보고서

## 2.1. 요구사항 분석

### 기능 요구사항

과제에서 제시한 최소 요구사항 7가지를 출발점으로 삼되, 그것만으로는 실제로 쓸 만한 서비스가
되지 않는다고 판단해 실제 중고거래 앱을 써보며 필요한 기능을 몇개 더 추가하였다.

| # | 최소 요구사항 | 실제로 구현한 세부 기능 |
|---|---|---|
| R1 | 사람들이 플랫폼에 가입할 수 있어야 함 | 회원가입/로그인/로그아웃, 아이디 중복 방지, 마이페이지(소개글과 비밀번호 변경), 사용자 프로필 조회, 2단계 인증|
| R2 | 상품들을 올리고 볼 수 있어야 함 | 상품 등록(상품명, 설명, 가격, 카테고리, 상품상태, 사진 5장), 목록/상세, 내 상품 관리, 조회수, 찜하기, 판매중/예약중/판매완료 상태 |
| R3 | 사용자들끼리 소통이 가능해야 함 | 실시간 전체 채팅, 1:1 채팅(내역 보존), 알림함 |
| R4 | 악성 유저나 상품을 차단해야 함 | 사유를 적어야 하는 신고, 신고 누적 시 자동 차단, 관리자 수동 처리 |
| R5 | 유저들 간의 송금이 가능해야 함 | 지갑과 충전, 송금에 더해 에스크로 안전거래, 가격 제안, 거래 후기, 일일 한도, PG 결제 충전 |
| R6 | 상품 검색이 가능해야 함 | 제목+설명 키워드 검색, 카테고리/가격대 필터, 정렬 |
| R7 | 관리자가 모든 요소를 관리할 수 있어야 함 | 사용자 휴면, 상품 차단과 삭제, 신고 처리, 에스크로 분쟁 중재, 재무 감사, 감사 로그 |

### 비기능 요구사항

- **보안**: OWASP Top 10 계열 취약점이 없도록 설계할 때부터 고려하였다.
- **신뢰성**: 결제와 송금, 이중 지불이나 잔액 음수, 거래 중 상품 삭제가 일어나면 안 됨
- **사용성**: 카드형 상품 그리드, 카테고리, 상태 뱃지 등 실제 서비스처럼 보이는 화면

### 요구사항이 바뀐 부분

만들어놓고 직접 써보니 요구사항 자체가 잘못됐다고 느낀 부분이 두 군데 있었다.
과제에서 논리적으로 타당하면 요구사항을 바꿔도 된다고 해서 다음과 같이 수정했다.

먼저 처음에는 "송금 기능"을 글자 그대로 구현해서 구매 버튼을 누르면 판매자에게 돈이 바로 갔다.
그런데 이러면 판매자가 물건을 안 보내도 구매자가 돈을 되찾을 방법이 없다. 실제 중고거래에서
사기가 제일 많이 일어나는 지점이기도 해서 대금을 플랫폼이 잠시 보관했다가 구매자가 물건을
받고 확정하면 그때 넘겨주는 에스크로 방식으로 요구사항을 바꿨다.

그다음으론 상품 목록과 상세를 전부 로그인 필수로 만들어두어서 등록된 상품을 누구나 볼 수 있어야 한다는 기준을 충족시키지 못했다.
요구사항 위반이라 비로그인 공개 열람으로 고쳤다.

## 2.2. 시스템 설계

### 전체 구조

```
[브라우저] ──HTTP──> [Flask 웹 서버 (app.py)] ──파라미터 바인딩──> [SQLite (market.db)]
     ├──WebSocket(Socket.IO)──> [Flask-SocketIO (실시간 채팅)] ──────┘
     └──결제창───────────────> [가상 PG / 토스페이먼츠] ──승인 콜백──> [금액 검증 후 크레딧]
```

단일 Flask 애플리케이션에 웹 라우트와 Socket.IO 이벤트 핸들러를 함께 두었고,
화면은 Jinja2 템플릿, 정적 자원은 `static/`에 두었다. Socket.IO 클라이언트 스크립트도
CDN 대신 로컬에 받아서 서빙하는데, 이건 CSP를 엄격하게 유지하기 위해서다.

### 웹페이지 설계

| 페이지 | 경로 | 설명 |
|---|---|---|
| 기본/가입/로그인 | `/`, `/register`, `/login`, `/login/2fa` | 인증. 2FA는 단계를 분리 |
| 홈 | `/dashboard` | 카테고리 칩 + 상품 카드 그리드 + 전체 채팅 |
| 검색 | `/search` | 키워드와 카테고리, 가격대, 정렬 |
| 상품 | `/product/new`, `/product/<id>`, `/product/<id>/edit`, `/product/<id>/delete` | 등록, 상세, 수정, 삭제 |
| 내 활동 | `/my/products`, `/my/favorites` | 내 상품 관리, 찜 목록 |
| 네고 | `/product/<id>/offer`, `/offer/<id>/respond` | 가격 제안과 응답 |
| 지갑 | `/wallet`, `/wallet/transfer` | 잔액, 송금, 거래내역, 에스크로 현황, 원장 |
| 충전(결제) | `/wallet/topup`, `/payment/checkout/<order_id>`, `/payment/mock/confirm`, `/payment/toss/success·fail` | 주문 생성, 결제창, 승인 |
| 에스크로 | `/product/<id>/buy`, `/escrow/<id>/confirm·cancel·approve_cancel·dispute·review` | 결제, 확정, 취소, 분쟁, 후기 |
| 소통 | `/chats`, `/chat/<username>`, `/notifications` | 1:1 채팅, 알림함 |
| 마이페이지 | `/profile`, `/profile/password`, `/profile/2fa/*` | 소개글과 비밀번호, 2FA |
| 사용자 조회 | `/user/<username>` | 프로필, 평점, 후기, 판매상품 |
| 신고 | `/report` | 사용자/상품 신고 |
| 관리자 | `/admin`, `/admin/users·products·reports·escrows·finance·logs` | 전체 관리, 분쟁 중재, 재무 감사 |

### 데이터베이스 설계

수업 자료의 예시는 사용자와 상품, 신고 세 개 테이블이었는데, 기능을 붙이다 보니 14개가 되었다.

```
user          (id PK, username UNIQUE, password_hash, bio, balance CHECK(>=0),
               is_admin, is_dormant, failed_logins, locked_until,
               totp_secret, session_ver, must_change_pw, created_at)
product       (id PK, title, description, price INT, category, condition,
               view_count, status[selling/reserved/sold] CHECK, is_blocked,
               seller_id FK, created_at)
product_image (id PK, product_id FK ON DELETE CASCADE, filename, sort)
favorite      (user_id FK, product_id FK, PK(user_id, product_id))
offer         (id PK, product_id FK, buyer_id FK, amount CHECK(>0),
               status[pending/accepted/rejected] CHECK, created_at)
escrow        (id PK, product_id FK, buyer_id FK, seller_id FK, amount CHECK(>0),
               status[held/released/refunded/cancel_requested/disputed] CHECK,
               created_at, updated_at)
payment       (id PK=order_id, user_id FK, amount CHECK(>0),
               status[pending/paid/failed/canceled] CHECK,
               provider[mock/toss], provider_key, created_at, updated_at)
ledger        (id PK, user_id FK, delta, balance_after, ref_type, ref_id, created_at)
review        (id PK, escrow_id FK, reviewer_id FK, target_id FK,
               rating CHECK(1~5), comment, UNIQUE(escrow_id, reviewer_id))
notification  (id PK, user_id FK, content, link, is_read, created_at)
report        (id PK, reporter_id FK, target_type CHECK, target_id, reason,
               status, UNIQUE(reporter_id, target_type, target_id))
message       (id PK, room, sender_id FK, content, created_at)
transfer      (id PK, sender_id FK, receiver_id FK, amount CHECK(>0),
               memo, kind[transfer/purchase], created_at)
audit_log     (id PK, user_id, action, detail, ip, created_at)
```

스키마를 짜면서 신경 쓴 부분은 다음과 같다.

- 비밀번호는 해시 컬럼으로만 두고 평문 컬럼 자체를 만들지 않았다.
- 금액은 전부 INTEGER에 `CHECK(>0)`, 잔액은 `CHECK(balance >= 0)`을 걸었다.
  애플리케이션 검증이 뚫려도 DB가 마지막으로 막아주게 하려는 의도다.
- 상태 값은 CHECK 제약으로 허용값을 못 박아서 이상한 값이 들어가지 못하게 했다.
- 신고와 후기는 UNIQUE 제약으로 중복을 DB 수준에서 막았다. 신고 폭탄 방지용이다.
- 기본키는 전부 UUIDv4다. 순차 번호면 남의 리소스를 번호만 바꿔가며 찔러볼 수 있기 때문이다.
- 1:1 채팅방 이름은 두 사용자 id를 정렬해서 서버가 만든다. 클라이언트가 방 이름을 정하면
  아무 방에나 들어갈 수 있게 된다.
- `ledger`는 모든 잔액 변동을 남기는 원장이다. 나중에 정합성을 검증할 수 있게 하려고 넣었다.

### 에스크로 상태 머신

에스크로는 돈이 걸린 부분이라 상태 전이를 먼저 그려놓고 구현했다.

```
        구매(결제: 구매자 잔액 차감, 상품 reserved)
                        │
                      [held] ──구매자 취소 요청──> [cancel_requested]
                     /      \                        /        \
        구매자 확정 /        \ 분쟁                판매자 승인   \ 분쟁
                  ▼           ▼                      ▼           ▼
             [released]   [disputed] ──관리자 중재──> released 또는 refunded
          (판매자 지급,   (관리자만 처리 가능)      [refunded] (구매자 환불,
           상품 sold)                                상품 selling 복귀)
```

각 전이마다 누가 할 수 있는지를 정해두고 검사한다. 구매 확정은 구매자만 취소 승인은
판매자만 중재는 관리자만 가능하다. 거래가 진행 중인 상품은 수정도 삭제도 중복 구매도 안 된다.
상태 전이는 조건부 UPDATE 한 문장으로 처리해서 같은 요청이 여러 번 들어와도 한 번만 정산된다.

### 결제 흐름

```
지갑에서 충전 금액 입력
  → POST /wallet/topup      : 주문 생성(금액을 서버 DB에 저장)
  → GET  /payment/checkout  : 결제창
  → 승인 콜백               : ① 주문 소유자인지 확인
                             ② 콜백으로 온 금액이 DB에 저장된 주문 금액과 같은지 확인
                             ③ 시크릿 키로 서버끼리 승인 확인
                             ④ pending→paid 조건부 UPDATE가 성공했을 때만 잔액 증가
```

금액을 클라이언트 말만 듣고 처리하지 않고 승인을 여러 번 받아도 한 번만 충전된다.

## 2.3. 시스템 구현

### 기술 선택

| 구성 | 선택 | 이유 |
|---|---|---|
| 웹 프레임워크 | Flask | 스켈레톤이 Flask라 그대로 유지 |
| 실시간 통신 | Flask-SocketIO | 전체/1:1 채팅 |
| DB | SQLite + 파라미터 바인딩 | 모든 쿼리를 `?` 바인딩으로 작성해 인젝션을 원천 차단 |
| CSRF | Flask-WTF (CSRFProtect) | 전역 적용이 간단함 |
| 비밀번호 해시 | bcrypt | 검증된 적응형 해시, salt 자동 처리 |
| 2FA | 표준 라이브러리로 TOTP 직접 구현 | 의존성을 늘리지 않기 위해. 비교는 `hmac.compare_digest`로 타이밍 공격 방지 |
| 결제 | 가상 PG + 토스페이먼츠 | 설정 없이도 시연되면서 실제 PG 연동 구조도 갖추기 위해 |

### 구현하면서 신경 쓴 부분

로그인은 5회 실패하면 5분 잠기고 여기에 IP당 5분 10회 제한을 더했다. 계정 잠금만 있으면
공격자가 계정을 바꿔가며 시도할 수 있기 때문이다. 2FA를 켠 계정은 비밀번호가 맞아도 세션을
발급하지 않고 코드 입력 단계로 넘어간다.

세션에는 `user.session_ver`를 같이 넣어두고 매 요청 비교한다. 비밀번호를 바꾸면 이 값이 올라가서
다른 기기에 남아 있던 세션이 전부 즉시 무효화된다.

잔액을 건드리는 코드는 전부 `apply_balance()` 하나만 지나가게 했다. 출금은
`UPDATE ... SET balance = balance + ? WHERE id = ? AND balance >= ?` 으로 검사와 차감을 동시에
하고 변동액과 변동 후 잔액을 원장에 남긴다. 에스크로도 대금이 판매자에게 바로 가지 않고 escrow
레코드로 보관되며 정산은 상태 조건부 UPDATE가 성공했을 때만 이뤄진다.

가격 제안은 판매가보다 낮아야 하고 구매자당 대기 중인 제안은 하나만 둘 수 있다. 수락과 거절은
판매자만 가능하다. 수락된 가격은 구매할 때 서버가 DB에서 다시 조회해서 쓴다. 결제도 주문 금액을
서버에 저장해두고 승인 콜백의 금액과 대조하며 주문은 소유자만 접근할 수 있다. 토스 모드라면
시크릿 키로 서버끼리 승인을 한 번 더 확인한다.

후기는 거래가 완료된 건에 대해서만 당사자만 한 번만 쓸 수 있다.

파일 업로드는 확장자 화이트리스트에 더해 파일 앞부분의 매직바이트를 실제로 읽어서 이미지가 맞는지
확인한다. 저장할 때는 원래 파일명을 버리고 랜덤 이름을 쓴다. 장수는 5장 용량은 10MB로 제한했다.

알림은 서버가 만든 문구와 링크만 저장한다.

### 발견한 보안 약점과 수정 내역

작업을 네 단계로 나눠 진행했기 때문에 찾은 순서대로 적었다. 전부 43건이다.

#### (1) 첫 코드에서 찾은 약점 (V1~V20)

받은 코드를 처음 읽으면서 찾은 것들이다.

| # | 보안 약점 | 위험 | 어떻게 고쳤나 |
|---|---|---|---|
| V1 | 비밀번호를 평문으로 저장하고 평문으로 비교 (`WHERE username=? AND password=?`) | DB가 유출되면 모든 계정이 그대로 털림 | bcrypt 해시+salt로 저장하고 `bcrypt.checkpw`로 비교 |
| V2 | `SECRET_KEY = 'secret!'` 하드코딩 | 세션 쿠키를 위조해 아무 계정이나 사칭 가능 | 환경변수 우선, 없으면 랜덤 32바이트 생성 후 `instance/secret_key`(0600)에 보관 |
| V3 | CSRF 보호가 아예 없음 | 로그인한 사용자를 속여서 송금 같은 요청을 실행시킬 수 있음 | Flask-WTF `CSRFProtect` 전역 적용, 모든 폼에 토큰 |
| V4 | 서버측 입력 검증이 없음 | XSS, 데이터 오염 | 정규식과 길이, 범위, 화이트리스트 검증을 전 입력에 적용 |
| V5 | 채팅에서 클라이언트가 보낸 `username`을 그대로 신뢰 | 다른 사람을 사칭해서 채팅 가능 | 서버 세션에서 사용자명을 조회해 사용 |
| V6 | Socket.IO에 인증이 없음 | 로그인 안 해도 채팅에 들어옴 | `connect`에서 세션 검증, 이벤트마다 다시 확인 |
| V7 | 채팅 rate limit 없음 | 도배와 스팸 | 사용자당 10초에 5건으로 제한 |
| V8 | `debug=True`로 실행 | 스택트레이스 노출은 물론 Werkzeug 디버거로 원격 코드 실행까지 가능 | `debug=False` + 커스텀 에러 페이지 |
| V9 | 세션 쿠키에 보호 플래그도 만료도 없음 | 세션 탈취 | HttpOnly, SameSite=Lax, (HTTPS면) Secure, 30분 만료 |
| V10 | 로그인 실패 횟수 제한 없음 | 무차별 대입 | 5회 실패 시 5분 계정 잠금 |
| V11 | 로그아웃이 GET | 링크만 클릭시켜도 강제 로그아웃 | POST + CSRF 토큰으로 변경 |
| V12 | `price`가 TEXT 컬럼 | 음수나 문자열 가격이 들어감 | INTEGER + CHECK + 서버 검증 |
| V13 | 리소스 접근 시 소유자 확인이 없음 | IDOR | 소유자/당사자 검증 후 아니면 403 |
| V14 | 신고 남용 방지가 없음 | 한 사람이 반복 신고해서 남을 차단시킬 수 있음 | UNIQUE 제약으로 1인 1대상 1회, 자기 자신과 관리자 신고 금지, 서로 다른 신고자 수로 집계 |
| V15 | 보안 헤더가 없음 | 클릭재킹, 스크립트 주입 | CSP(nonce 기반), X-Frame-Options, nosniff, Referrer-Policy |
| V16 | 세션 고정 공격 대응 없음 | 공격자가 심어둔 세션으로 로그인시켜 탈취 | 로그인 성공 시 `session.clear()` 후 재발급 |
| V17 | LIKE 검색에서 와일드카드를 이스케이프하지 않음 | LIKE 인젝션 | `%`, `_`, `\`를 이스케이프한 뒤 바인딩 |
| V18 | 파일 업로드 검증 없음 | 웹셸 업로드 | 확장자 화이트리스트 + 매직바이트 + 랜덤 파일명 + 용량 제한 |
| V19 | 송금에 동시성 대응이 없음 | 이중 지불, 잔액 음수 | 조건부 원자적 UPDATE + CHECK 제약 |
| V20 | 감사 추적이 불가능 | 사고가 나도 누가 뭘 했는지 알 수 없음 | `audit_log` 테이블에 주요 행위 기록 |

#### (2) 실거래 기능을 붙이면서 찾은 약점 (V21~V30)

기능을 확장하니 새로운 위협이 따라왔다.

| # | 약점/위협 | 위험 | 어떻게 고쳤나 |
|---|---|---|---|
| V21 | 직접 송금 방식이라 판매자가 물건을 안 보내면 구제가 안 됨 | 거래 사기 | 에스크로 도입. 대금을 플랫폼이 보관하고 구매 확정 시 지급, 분쟁은 관리자가 중재 |
| V22 | 비밀번호가 유출되면 그걸로 끝 | 계정 탈취 | TOTP 2단계 인증 추가 |
| V23 | 비밀번호를 바꿔도 이미 탈취된 세션은 계속 살아 있음 | 탈취 지속 | 세션 버전 도입, 변경 시 전 기기 세션 무효화 |
| V24 | 계정이 털리면 잔액 전부가 한 번에 빠져나감 | 피해 확대 | 일일 거래 한도로 상한을 둠 |
| V25 | 계정 잠금만 있으면 계정을 바꿔가며 공격 가능 | 무차별 대입 | IP 단위 로그인 rate limit 추가 |
| V26 | 관리자 기본 비밀번호를 안 바꾸고 방치할 수 있음 | 관리자 탈취 | 기본 비밀번호로 생성됐으면 첫 로그인에서 변경 강제 |
| V27 | 에스크로와 제안 상태를 아무나 바꿀 수 있으면 자금 탈취로 이어짐 | 권한 상승 | 상태 머신 + 행위자 검증, E2E 테스트로 확인 |
| V28 | 수락된 제안 가격을 클라이언트가 위조할 수 있음 | 가격 조작 | 결제 금액은 서버가 DB에서 다시 조회해서 사용 |
| V29 | HTTPS 환경인데 HSTS가 없음 | SSL 스트리핑 | `SESSION_COOKIE_SECURE=1`이면 HSTS 헤더 자동 적용 |
| V30 | 정렬과 카테고리 파라미터를 쿼리에 그대로 쓰면 인젝션 | SQL 인젝션 | 화이트리스트에 매핑해서 사용하고, 벗어나면 400 |

#### (3) 금융 쪽을 강화하면서 찾은 약점 (V31~V37)

실제로 자산이 오가는 수준으로 만들다 보니, 일반 웹 취약점과는 결이 다른 문제들이 보였다.
특히 V31은 동시성 테스트를 만들어 돌려보고 나서야 실제로 재현된 문제였다.

| # | 약점/위협 | 위험 | 어떻게 고쳤나 |
|---|---|---|---|
| V31 | 에스크로 정산이 `상태 확인 → 잔액 지급` 두 단계라 동시에 요청하면 이중 지급 가능 | 자금이 두 번 나감 | 상태 조건부 UPDATE 한 문장으로 바꾸고 `rowcount==1`일 때만 지급. 동시에 20번 확정 요청해도 정확히 한 번만 정산되는 것을 테스트로 확인 |
| V32 | 잔액 변동이 `user.balance`에만 남아 사후 검증이 불가능 | 내부 부정이나 버그를 못 잡음 | `apply_balance()` 단일 통로 + 이중기입. 관리자 화면에서 원장 합계와 실제 잔액을 대조 |
| V33 | 송금과 결제 폼을 더블클릭하거나 새로고침하면 중복 거래 | 의도치 않은 중복 이체 | 폼마다 1회용 토큰을 발급하고, 쓴 토큰은 재사용 불가 |
| V34 | 비밀번호만으로 송금이 승인됨 | 비밀번호 유출 시 즉시 자금 이동 | 2FA를 켠 사용자는 거래할 때 TOTP 코드도 요구 |
| V35 | 탈취 계정으로 짧은 시간에 여러 번, 고액으로 빼갈 수 있음 | 대량 유출 | 간단한 이상거래탐지. 10분에 5회 초과 출금 차단 100만원 이상은 관리자에게 알림 |
| V36 | 음수 잔액을 DB에서 막는 최종 방어선이 없음 | 무결성 훼손 | `balance CHECK(>= 0)` 제약 추가 |
| V37 | 결제 가능 여부를 애플리케이션 로직에만 의존 | 잔액 초과 결제 | 차감을 `balance >= ?` 조건부 UPDATE로만 수행 (V19 원칙을 전 거래 경로에 일관 적용) |

이 과정에서 잔액을 건드리는 코드는 한 함수만 지나가게 하고 검사와 차감은 한 SQL 문 안에서
끝내는 쪽으로 방향을 잡았다. 상태 전이는 조건부 UPDATE로 멱등하게 만들고 변동은 전부 원장에
남겼다.

#### (4) PG 결제를 붙이면서 찾은 약점 (V38~V43)

충전을 "버튼 누르면 잔액 증가"에서 실제 결제 흐름으로 바꾸자 결제 도메인 특유의 취약점이 생겼다.

| # | 약점/위협 | 위험 | 어떻게 고쳤나 |
|---|---|---|---|
| V38 | 승인 콜백의 금액을 클라이언트가 위조 | 부정 충전 | 콜백 금액이 서버에 저장된 주문 금액과 정확히 같을 때만 처리 |
| V39 | 승인 콜백이 중복으로 오거나 새로고침하면 이중 충전 | 부정 충전 | `pending→paid` 조건부 UPDATE로 멱등 처리. 동시 20회 승인에도 한 번만 충전됨을 테스트로 확인 |
| V40 | 남의 주문번호로 접근해서 승인 시도 | IDOR, 부정 충전 | 주문 조회 시 소유자 검증, 주문번호는 UUID |
| V41 | PG가 성공했다는 클라이언트 응답만 믿고 처리 | 위조된 성공 응답 | 토스는 시크릿 키로 서버끼리 승인 확인. 시크릿은 클라이언트에 절대 노출하지 않음 |
| V42 | 결제창 때문에 외부 스크립트를 허용하면 CSP가 약해짐 | XSS 방어 약화 | 토스 도메인은 `PAYMENT_PROVIDER=toss`일 때만 최소 범위로 허용하고, 기본값에서는 엄격하게 유지 |
| V43 | 결제 주문 폼 더블클릭으로 중복 주문 | 중복 결제 | 주문 생성에도 1회용 폼 토큰 적용 |

## 2.4. 체크리스트 작성 및 테스팅

### 보안 체크리스트 점검

항목마다 실제로 구현됐는지와 무엇으로 확인했는지를 적었다. 완전히 못 지킨 항목은 △로 표시하고 이유를 남겼다.

### 테스트 실행과 결과

e2e는 가입부터 로그인, 상품 등록, 비로그인 열람, 검색, 찜, 가격 제안과 수락, 에스크로 결제,
구매 확정, 후기, 취소와 환불, 신고 누적에 따른 자동 차단과 관리자 처리와 분쟁 중재, 알림, PG 충전까지 한 번에 훑는다.
여기에 CSRF,SQL 인젝션,XSS,IDOR 같은 공격 시도를 섞어두었다.

chat은 Socket.IO 프로토콜을 직접 주고받는다. 두 사용자를 접속시켜 전체 채팅이 실제로
상대에게 도착하는지, 1:1 메시지가 제3자에게는 새지 않는지 새로고침 후에도 내역이 남는지를
확인한다.

finance는 동시성 문제를 잡으려고 만든 스크립트다. 같은 에스크로에 구매확정 요청을 20개
동시에 던지고 판매자 잔액이 딱 한 번만 늘었는지, 원장 기록이 1건인지를 DB에서 직접 조회해
확인한다.

## 2.5. 유지보수

### 만들고 나서 고친 것들

한 번에 완성한 게 아니라 만들고 써보고 문제를 발견해서 다시 고치는 과정을 여러 번 반복했다.
구현 과정에서의 시행 착오를 적어두었다.

**직접 써보고 발견한 문제**

| 발견한 문제 | 어느 단계의 문제인가 | 어떻게 고쳤나 |
|---|---|---|
| 직접 송금 구매는 먹튀에 무방비 | 요구사항 자체가 잘못됨 | 에스크로로 요구사항을 바꾸고 재설계 |
| 정찰제만 있어서 실제 중고거래 관행과 안 맞음 | 요구사항 누락 | 가격 제안 기능 추가 |
| 판매자를 믿을 수 있는지 알 방법이 없음 | 요구사항 누락 | 거래 후기와 평점 추가 |
| 상품이 늘어나니 목록만으로 못 찾음 | 설계 한계 | 카테고리와 필터, 정렬, 찜 추가 |
| 거래가 진행돼도 당사자가 모름 | 설계 한계 | 알림함 추가 |
| 채팅이 새로고침하면 사라짐 | 구현 이슈 | 채팅 내역을 DB에 저장하고 불러오게 수정 |
| 신고할 때 UUID를 직접 입력해야 함 | 사용성 | 대상이 미리 채워진 신고 링크 제공 |

**테스트를 돌리다가 발견한 문제**

- 일일 한도를 넣었더니 기존 "잔액 초과" 테스트가 한도 검사에 먼저 걸려서 엉뚱하게 통과했다.
  테스트를 한도 케이스와 잔액 케이스로 분리했다.
- IP rate limit이 자동화 테스트와 충돌해서 로그인이 막혔다. `IP_LOGIN_LIMIT`을
  환경변수로 조정할 수 있게 바꿨다.
- 동시성 테스트를 만들어 돌려보니 에스크로 정산에서 이중 지급이 실제로 재현됐다.
  이게 V31이고, 상태 조건부 UPDATE로 다시 작성했다. 테스트를 안 만들었으면 못 찾았을 문제다.
- 폼 토큰을 넣은 뒤 "예약중 상품 재구매" 테스트가 실패했다. 확인해보니 예약중 상품은
  구매 폼 자체가 렌더링되지 않아서 토큰을 얻을 수 없는, 오히려 정상적인 이중 방어였다.
  테스트를 두 경로 모두 통과로 인정하도록 수정했다.

**요구사항을 다시 읽어보고 발견한 문제**

- "등록된 상품은 누구나 볼 수 있어야 함"인데 로그인 필수로 되어 있었다.
  공개 열람으로 바꾸고, 대신 구매와 채팅, 찜 같은 액션은 로그인이 필요하도록 유지했다.
  비로그인 상태로 액션 엔드포인트를 직접 찔러도 막히는지 E2E에 케이스를 추가했다.
- 로컬에서 시연하기가 번거로워서 `seed.py`를 만들었다. 상품과 계정, 사진, 잔액을
  한 번에 넣어준다.
