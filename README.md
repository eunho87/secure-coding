# Tiny Second-hand Shopping Platform (Secure Coding)

시큐어 코딩을 적용한 중고거래 플랫폼 **MyMarket** 과 그 **개발 전 과정 보고서**입니다.
Flask + Flask-SocketIO + SQLite 기반으로, 실제 중고거래 서비스(당근마켓·번개장터 등)를 벤치마킹해
에스크로 안전거래·가격 제안(네고)·거래 후기·PG 결제 충전까지 실거래 수준으로 구현했습니다.

| 항목 | 내용 |
|---|---|
| GitHub | https://github.com/eunho87/secure-coding |
| 개발 환경 | WSL2 Ubuntu, Python 3.9(conda) / 3.12(venv), Flask 3.x |
| 기반 코드 | https://github.com/ugonfor/secure-coding (스켈레톤 clone 후 확장) |
| 개발 방식 | 4회 반복(iteration) — v1 최소요구+보안 → v2 실거래 기능 → v3 금융 보안 강화 → v4 PG 결제 연동 |
| 테스트 | 자동화 81개 전부 PASS (e2e 67 + extra 8 + finance 6) |
| 발견·수정한 보안 약점 | **43건 (V1~V43)** — [3.3장](#33-발견한-보안-약점과-수정-내역-v1v43) |

> 추가 실행 절차(외부 공개·실결제 키·배포 등 개인 계정이 필요한 작업)는 **[실행방법.md](실행방법.md)** 에 별도로 정리했습니다.

# 빠른 시작

## 환경 설정

Ubuntu(WSL 포함) + Python 3.9 이상 기준입니다. 아래 둘 중 하나를 선택하세요.

### 방법 1: conda (miniconda/anaconda)

```bash
git clone https://github.com/eunho87/secure-coding
cd secure-coding
conda env create -f enviroments.yaml
conda activate secure_coding
```

### 방법 2: venv + pip

```bash
git clone https://github.com/eunho87/secure-coding
cd secure-coding
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## 실행

```bash
python app.py
```

- 서버 주소: **http://127.0.0.1:5000**
- 최초 실행 시 `market.db`(SQLite)가 자동 생성됩니다.
- 상품 목록·상세·검색은 **로그인 없이 누구나** 열람 가능(요구사항), 구매·채팅·찜·판매 등은 로그인 필요.

## 데모 데이터 등록 (채점·시연 권장)

빈 화면 대신 상품·계정·사진·잔액을 미리 채워, 바로 "사고 파는" 과정을 볼 수 있습니다.
서버가 DB를 만든 뒤(한 번 실행 후) 다른 터미널에서 실행하세요.

```bash
python seed.py          # 데모 상품 15개 + 계정 4개 + 사진 등록
```

- **데모 계정 (비밀번호 공통 `Passw0rd1`)**
  - 판매자: `daangn_kim`, `market_lee`, `hanguk_park`
  - 구매자: `buyer_choi` (잔액 넉넉 — 바로 구매 시연 가능)
- 시연 순서는 [4.5 데모 시나리오](#45-데모-시나리오-수동-시연-절차) 참고.

## 관리자 계정

최초 실행 시 자동 생성됩니다.

- 아이디: `admin`
- 비밀번호: 환경변수 `ADMIN_PASSWORD` 값 (미지정 시 `Admin123!`)
- **기본 비밀번호로 생성되면 최초 로그인 시 비밀번호 변경이 강제됩니다.**

```bash
# 권장: 실행 전 관리자 비밀번호 지정
ADMIN_PASSWORD='나만의강력한비밀번호1' python app.py
```

## 테스트 실행

기능·보안·금융을 실제 HTTP 요청/DB 조회로 검증하는 자동화 스크립트입니다.
로그인 IP rate limit이 자동화와 충돌하지 않도록 서버를 `IP_LOGIN_LIMIT=1000`으로 띄웁니다.

```bash
IP_LOGIN_LIMIT=1000 python app.py &     # 테스트용 서버 실행
bash tests/e2e_test.sh      # 기능 + 보안 E2E (67개)
bash tests/extra_test.sh    # 소켓 인증·업로드 검증·2FA (8개)
bash tests/finance_test.sh  # 금융: 이중정산·결제멱등·원장무결성 (6개)
```

**총 81개 전부 PASS.** 상세 케이스는 [4장](#4-체크리스트-작성-및-테스팅) 참고.

# 개발 전 과정 보고서

## 1. 요구사항 분석

### 1.1 기능 요구사항

과제의 최소 요구사항을 기준으로, 실제 중고거래 서비스를 벤치마킹하여 세부 요구사항을 도출했다.

| # | 최소 요구사항 | 도출한 세부 요구사항 (v1 → v2~v4 확장) |
|---|---|---|
| R1 | 사람들이 플랫폼에 가입할 수 있어야 함 | 회원가입/로그인/로그아웃, 아이디 중복 방지, 마이페이지(소개글·비밀번호 변경), 사용자 프로필 조회, **2단계 인증(TOTP)** |
| R2 | 상품들을 올리고 볼 수 있어야 함 | 상품 등록(상품명·설명·가격·**카테고리·상품상태·사진 5장**), 목록/상세 페이지, 내 상품 관리, **조회수·찜하기**, 판매중/예약중/판매완료 상태 |
| R3 | 사용자들끼리 소통이 가능해야 함 | 실시간 전체 채팅, 1:1 채팅(내역 보존), **알림함** |
| R4 | 악성 유저나 상품을 차단해야 함 | 사용자/상품 신고(사유 필수), 신고 누적 자동 차단(상품 3회)/휴면(유저 5회), 관리자 수동 처리 |
| R5 | 유저들 간의 송금이 가능해야 함 | 지갑/충전/송금 + **에스크로 안전거래**(대금 보관→구매확정 시 지급, 취소/분쟁 중재), **가격 제안(네고)**, **거래 후기/평점**, 일일 거래 한도, **PG 결제 충전** |
| R6 | 상품 검색이 가능해야 함 | 키워드(제목+설명) + **카테고리/가격대 필터, 정렬(최신·저가·고가·인기)** |
| R7 | 관리자가 모든 요소를 관리할 수 있어야 함 | 사용자(휴면)/상품(차단·삭제)/신고 처리, **에스크로 분쟁 중재(지급/환불)**, **재무 감사**, 감사 로그 |

### 1.2 비기능 요구사항

- **보안**: 과제의 핵심. OWASP Top 10 계열 취약점이 없도록 설계 단계부터 반영 (3장, 4장)
- **신뢰성**: 결제·송금은 원자적으로 처리되어 이중 지불·잔액 음수·거래 중 상품 삭제가 불가능해야 함
- **사용성**: 카드형 상품 그리드, 카테고리 칩, 상태 뱃지 등 실서비스 수준의 화면 구성

### 1.3 요구사항 변경 이력 (유지보수 반영)

v1 사용 평가에서 "직접 송금 기반 구매는 사기에 취약하다(먹튀 가능)"는 문제를 확인
→ **에스크로 방식으로 요구사항을 변경**하고 v2에서 구현했다.
(과제 조건: 논리적으로 타당하다면 요구사항 변경 가능)

또한 "등록된 상품은 누구나 볼 수 있어야 함" 요구사항 재점검 결과 로그인 필수로 구현되어 있어
**비로그인 공개 열람으로 수정**했다 (5장 참고).

## 2. 시스템 설계

### 2.1 아키텍처

```
[브라우저] ──HTTP──> [Flask 웹 서버 (app.py)] ──파라미터 바인딩──> [SQLite (market.db)]
     ├──WebSocket(Socket.IO)──> [Flask-SocketIO (실시간 채팅)] ──────┘
     └──결제창(PG)──────────> [가상 PG / 토스페이먼츠] ──승인 콜백──> [서버 금액검증·멱등 크레딧]
```

### 2.2 웹페이지 설계

| 페이지 | 경로 | 설명 |
|---|---|---|
| 기본/가입/로그인 | `/`, `/register`, `/login`, `/login/2fa` | 인증 (2FA 단계 분리) |
| 홈(대시보드) | `/dashboard` | 카테고리 칩 + 상품 카드 그리드 + 전체 채팅 |
| 검색 | `/search` | 키워드/카테고리/가격대/정렬 필터 |
| 상품 | `/product/new`, `/product/<id>`, `/product/<id>/edit`, `/product/<id>/delete` | 등록(다중 사진)/상세(갤러리·찜·네고·안전거래)/수정/삭제 |
| 내 활동 | `/my/products`, `/my/favorites` | 내 상품 관리, 찜 목록 |
| 네고 | `/product/<id>/offer`, `/offer/<id>/respond` | 가격 제안/응답 |
| 지갑 | `/wallet`, `/wallet/transfer` | 잔액·송금·거래내역·에스크로 현황·원장 |
| 결제(충전) | `/wallet/topup`, `/payment/checkout/<order_id>`, `/payment/mock/confirm`, `/payment/toss/success·fail` | 주문 생성 → 결제창 → 승인 콜백 |
| 에스크로 | `/product/<id>/buy`, `/escrow/<id>/confirm·cancel·approve_cancel·dispute·review` | 결제→확정/취소/분쟁/후기 |
| 소통 | `/chats`, `/chat/<username>`, `/notifications` | 1:1 채팅, 알림함 |
| 마이페이지 | `/profile`, `/profile/password`, `/profile/2fa/*` | 소개글·비밀번호·2FA 관리 |
| 사용자 조회 | `/user/<username>` | 프로필·평점·후기·판매상품 |
| 신고 | `/report` | 사용자/상품 신고 |
| 관리자 | `/admin`, `/admin/users·products·reports·escrows·finance·logs` | 전체 관리 + 분쟁 중재 + 재무 감사 |

### 2.3 데이터베이스 설계 (14개 테이블)

과제 슬라이드의 기본 3개 테이블(사용자·상품·신고)에서 출발해, 기능 확장에 따라 14개로 설계했다.

```
user          (id PK, username UNIQUE, password_hash, bio, balance CHECK(>=0),
               is_admin, is_dormant, failed_logins, locked_until,
               totp_secret, session_ver, must_change_pw, created_at)
product       (id PK, title, description, price INT, category, condition,
               view_count, status[selling/reserved/sold] CHECK, is_blocked,
               seller_id FK, created_at)
product_image (id PK, product_id FK ON DELETE CASCADE, filename, sort)
favorite      (user_id FK, product_id FK, PK(user_id, product_id))
offer         (id PK, product_id FK, buyer_id FK, amount CHECK(>0),
               status[pending/accepted/rejected] CHECK, created_at)
escrow        (id PK, product_id FK, buyer_id FK, seller_id FK, amount CHECK(>0),
               status[held/released/refunded/cancel_requested/disputed] CHECK,
               created_at, updated_at)
payment       (id PK=order_id, user_id FK, amount CHECK(>0),
               status[pending/paid/failed/canceled] CHECK,
               provider[mock/toss], provider_key, created_at, updated_at)
ledger        (id PK, user_id FK, delta, balance_after, ref_type, ref_id, created_at)
review        (id PK, escrow_id FK, reviewer_id FK, target_id FK,
               rating CHECK(1~5), comment, UNIQUE(escrow_id, reviewer_id))
notification  (id PK, user_id FK, content, link, is_read, created_at)
report        (id PK, reporter_id FK, target_type CHECK, target_id, reason,
               status, UNIQUE(reporter_id, target_type, target_id))
message       (id PK, room, sender_id FK, content, created_at)
transfer      (id PK, sender_id FK, receiver_id FK, amount CHECK(>0),
               memo, kind[transfer/purchase], created_at)
audit_log     (id PK, user_id, action, detail, ip, created_at)
```

설계 시 보안 고려사항:

- 비밀번호는 해시 컬럼으로만 저장, 금액은 INTEGER + CHECK 제약, 잔액은 `CHECK(balance >= 0)`
- 상태 값은 전부 CHECK 제약(화이트리스트)으로 DB 수준에서 강제
- 신고·후기는 UNIQUE 제약으로 중복(남용) 방지
- PK는 추측 불가능한 UUIDv4 (순차 ID 열거 공격 방지)
- 1:1 채팅방 이름은 서버가 두 사용자 id를 정렬해 생성 (임의 방 접근 차단)
- `ledger`는 모든 잔액 변동의 이중기입 원장 → 사후 무결성 감사 가능

### 2.4 에스크로 상태 머신 설계

```
        구매(결제: 구매자 잔액 차감, 상품 reserved)
                        │
                      [held] ──구매자 취소 요청──> [cancel_requested]
                     /      \                        /        \
        구매자 확정 /        \ 분쟁                판매자 승인   \ 분쟁
                  ▼           ▼                      ▼           ▼
             [released]   [disputed] ──관리자 중재──> released 또는 refunded
          (판매자 지급,   (관리자만 처리 가능)      [refunded] (구매자 환불,
           상품 sold)                                상품 selling 복귀)
```

- 모든 상태 전이는 **행위자 권한 검증**(구매자만 확정, 판매자만 취소 승인, 관리자만 중재) 후 수행
- 거래 진행 중(reserved) 상품은 수정/삭제/중복 구매 불가
- 상태 전이는 **조건부 단일 UPDATE**로 멱등 처리 (동시 요청에도 1회만 정산 — V31)

### 2.5 결제(PG) 흐름 설계

```
[지갑] 충전 금액 입력
   → POST /wallet/topup       : payment 주문 생성(status=pending, 금액을 서버에 저장)
   → GET  /payment/checkout   : 결제창 (mock 가상 PG 또는 토스 위젯)
   → 승인 콜백                 : ① 주문 소유자 검증 ② 금액이 서버 저장값과 일치하는지 검증
                                ③ (토스) 시크릿 키로 서버-서버 승인 확인
                                ④ pending→paid 조건부 UPDATE 성공 시에만 잔액 크레딧(멱등)
```

## 3. 시스템 구현

### 3.1 구현 스택

| 구성 | 선택 | 이유 |
|---|---|---|
| 웹 프레임워크 | Flask | 스켈레톤 유지, 경량 |
| 실시간 통신 | Flask-SocketIO | 전체/1:1 채팅 |
| DB | SQLite + 파라미터 바인딩 | 모든 쿼리 `?` 바인딩으로 인젝션 원천 차단 |
| CSRF | Flask-WTF (CSRFProtect) | 전역 CSRF 보호 |
| 비밀번호 해시 | bcrypt | 적응형 해시, salt 자동 |
| 2FA | 표준 라이브러리로 RFC 6238 TOTP 직접 구현 | 의존성 최소화, `hmac.compare_digest`로 타이밍 공격 방지 |
| 결제 | 가상 PG(기본) / 토스페이먼츠 | 무설정 시연 + 실제 PG 연동 구조 동시 충족 |
| 배포 | gunicorn + gevent-websocket | WebSocket 지원 프로덕션 서버 |

### 3.2 기능별 핵심 구현 포인트

- **인증**: 로그인 5회 실패 → 5분 잠금 + IP당 5분 10회 rate limit. 2FA 활성 계정은
  비밀번호 검증 후에도 세션을 발급하지 않고 별도 2FA 단계(시도 5회 제한)를 통과해야 로그인 완료
- **세션 버전**: `user.session_ver`를 세션에 함께 저장하고 매 요청 비교.
  비밀번호 변경 시 버전 증가 → 탈취된 세션 포함 다른 기기 세션 전부 즉시 무효화
- **잔액 처리**: 모든 잔액 변동은 `apply_balance()` 단일 통로만 통과.
  출금은 `UPDATE ... SET balance = balance + ? WHERE id = ? AND balance >= ?` 단일 문으로
  검사+차감을 원자화하고, 변동액·변동 후 잔액을 원장에 기록
- **에스크로**: 대금은 판매자에게 바로 가지 않고 escrow 레코드로 보관되며,
  정산(지급/환불)은 상태 조건부 UPDATE가 성공한 경우에만 수행 (멱등)
- **가격 제안**: 판매가 미만 검증, 구매자당 1건의 pending 제안, 판매자만 수락/거절 가능(403).
  수락된 제안가는 구매 시 서버가 다시 조회해 적용 (클라이언트가 가격을 보내지 않음)
- **결제**: 주문 금액을 서버에 저장하고 승인 콜백의 금액과 대조, 소유자 검증(404),
  토스는 시크릿 키로 서버-서버 승인, 1회용 폼 토큰으로 중복 주문 차단
- **후기**: 거래 완료(released) 상태에서만, 거래 당사자만, 거래당 1회(UNIQUE 제약)
- **파일 업로드**: 확장자 화이트리스트 + 매직바이트 검증 + 랜덤 파일명 + 장수(5)/용량(10MB) 제한,
  업로드 파일 서빙 경로는 파일명 형식 검증(정규식) 후 인증 사용자에게만 제공
- **알림**: 서버가 생성한 내용/링크만 저장 (사용자 입력 미포함)

#### (3) v3 금융 거래 보안 강화에서 식별·보완 (V31~V37)

실제 자산이 오가는 거래 시스템으로 발전시키면서, 금융 도메인 특유의 위협을 집중 보완했다.

| # | 식별한 약점/위협 | 위험 | 보완 내용 |
|---|---|---|---|
| V31 | 에스크로 정산이 `상태 확인 → UPDATE` 2단계라 동시 요청 시 **이중 지급** 가능 (TOCTOU race) | 자금 이중 인출 | 정산을 **상태 조건부 단일 UPDATE**로 원자화. `rowcount==1`일 때만 지급/환불. 동시 confirm 20회에도 정확히 1회만 정산됨을 테스트로 검증 |
| V32 | 잔액 변동이 `user.balance`에만 반영되어 **감사·정합성 검증 불가** | 내부 부정·오류 탐지 불가 | 단일 함수 `apply_balance()` 도입 + **이중기입 원장(ledger)** 기록. 관리자 재무 감사에서 원장 합계 vs 실제 잔액 대조 |
| V33 | 송금/결제 폼 **더블클릭·새로고침 재전송** 시 중복 거래 | 의도치 않은 중복 이체 | 거래 폼마다 **1회용 서버 토큰**(form_token) 발급, 소비 후 재사용 차단 |
| V34 | 비밀번호만으로 송금·결제 승인 → 비밀번호 유출 시 즉시 자금 이동 | 자금 탈취 | 2FA 활성 사용자는 **거래 시 TOTP 코드 추가 요구** |
| V35 | 탈취 계정으로 **단시간 다수 출금**(velocity) 및 고액 이체 무제한 | 대량 자금 유출 | **이상거래탐지(FDS)**: 10분 내 5회 초과 출금 차단, 고액(100만원↑) 거래는 관리자 자동 알림 |
| V36 | DB 레벨에서 음수 잔액을 막는 최종 방어선 부재 | 데이터 무결성 훼손 | `user.balance`에 **`CHECK(balance >= 0)` 제약** 추가 |
| V37 | 결제 금액을 애플리케이션 로직에만 의존 | 잔액 초과 결제 | 잔액 차감을 `balance >= ?` 조건부 UPDATE로만 수행 (V19 원칙을 전 거래 경로에 일관 적용) |

**금융 보안 설계 원칙**: ① 모든 잔액 변동은 `apply_balance()` 단일 통로 ② 잔액 검사와 차감은
**한 SQL 문 안에서** 원자적으로 ③ 상태 전이는 조건부 UPDATE로 멱등하게 ④ 모든 변동은 원장에 기록
⑤ 애플리케이션·DB 이중 방어.

#### (4) v4 실제 PG(결제) 연동에서 식별·보완 (V38~V43)

지갑 충전을 "버튼 누르면 즉시 잔액 증가"에서 **실제 PG 결제 흐름**(주문 생성 → 결제창 →
승인 콜백)으로 바꾸면서, 결제 도메인의 대표적 공격을 방어했다.

| # | 식별한 약점/위협 | 위험 | 보완 내용 |
|---|---|---|---|
| V38 | 승인 콜백의 **금액을 클라이언트가 위·변조**(100원 결제 후 100만원 크레딧 요청) | 부정 충전 | 콜백 금액이 **서버에 저장된 주문 금액과 정확히 일치**할 때만 처리 |
| V39 | 결제 승인 콜백 **중복 수신/새로고침**으로 이중 충전 | 부정 충전 | `pending→paid` 조건부 UPDATE로 **멱등 처리** (동시 20회 승인에도 1회만 크레딧 — 테스트 검증) |
| V40 | 타인의 결제 주문(order_id)에 접근·승인 | IDOR·부정 충전 | 주문 조회 시 **소유자 검증**(본인 주문 아니면 404), 주문 ID는 UUID |
| V41 | PG 승인을 클라이언트 응답만 믿고 처리 | 위조된 성공 응답 | 토스 결제는 **시크릿 키로 서버-서버 확인 API** 호출로 최종 승인 (시크릿 비노출) |
| V42 | 결제창(외부 스크립트) 도입으로 CSP 무력화 우려 | XSS 방어 약화 | 토스 도메인은 `PAYMENT_PROVIDER=toss`일 때만 CSP에 **최소 범위 허용**, 기본은 strict 유지 |
| V43 | 결제 주문 폼 **더블클릭**으로 중복 주문 | 중복 결제 | 주문 생성에도 1회용 폼 토큰 적용 (V33 확장) |

### 3.4 잔여 위험 및 한계 (정직한 평가)

  → `Dockerfile`/`Procfile`/`render.yaml`로 제공, [실행방법.md](실행방법.md) 5장 명시
- rate limit이 프로세스 메모리 기반 → 다중 프로세스 배포 시 Redis 등으로 교체 필요
- 충전은 기본이 가상 PG(모의 재화). 실 결제 운영에는 PG 심사·본인확인(KYC)·정식 FDS 고도화 필요
  (현재 FDS는 velocity/고액 알림 수준의 기본형)
- 2FA 복구 코드(백업 코드) 미구현 → 인증 앱 분실 시 관리자 개입 필요
- 원장·잔액이 단일 SQLite 트랜잭션으로 정합성을 유지하나, 분산 배포 시 트랜잭션 격리·분산 락 검토 필요

### 4.5 자동화 실행 및 최종 결과

```bash
IP_LOGIN_LIMIT=1000 python app.py &
bash tests/e2e_test.sh      # 67개
bash tests/extra_test.sh    #  8개
bash tests/finance_test.sh  #  6개
```

- **기능**: 가입→로그인→상품 등록(카테고리/사진)→**비로그인 공개 열람**→검색(필터/정렬)→찜→
  가격 제안→수락→에스크로 결제→구매 확정(판매자 잔액 정확성 검증)→후기→취소 흐름
  (요청→승인→환불)→신고→관리자 처리→알림 수신→**PG 결제 충전** 전체 시나리오
- **일반 보안**: CSRF 차단, SQL 인젝션, XSS 이스케이프, IDOR(상품/제안/에스크로/결제),
  계정 잠금, 미인증 소켓·구매·판매 차단, 가짜 이미지 차단, 2FA 전체 흐름,
  관리자 초기 비밀번호 변경 강제, 에러 페이지 정보 비노출
- **금융 보안**: 1회용 폼 토큰 이중 제출 차단, 일일 한도, 잔액 초과 차단, 원장 표시,
  관리자 재무 무결성 감사, **에스크로 동시 confirm 20회 → 1회만 정산**,
  **결제 승인 20회 동시 요청 → 1회만 크레딧**, 결제 주문 IDOR 차단, 원장-잔액 정합성,
  음수 잔액 불가

**최종 결과: 전 항목 PASS (81/81)**
프로덕션 이미지(Docker + gunicorn)에서도 e2e 67 + extra 8 = 75개 통과를 확인했다.

### 4.6 데모 시나리오 (수동 시연 절차)

시드 데이터(`python seed.py`)가 등록된 상태에서 아래 순서로 "사고 파는" 전 과정을 시연할 수 있다.
데모 계정 비밀번호는 모두 `Passw0rd1`.

1. **(비로그인) 상품 구경** — 첫 화면 → "상품 구경하기" → 15개 상품을 로그인 없이 열람 (TC-P06)
2. **buyer_choi 로그인** — 잔액 3,000,000원 보유
3. **아이폰 13 상세** → 등록된 500,000원 가격 제안 확인 (판매자 수락 시 그 가격으로 구매)
4. **다른 상품 안전거래 구매** — 예: "닌텐도 스위치" → 비밀번호 입력 → 결제 → 대금 에스크로 보관
5. **지갑 → 안전거래 현황** — 구매 확정 → 판매자 지급 → 거래 후기(별점) 작성
6. **hanguk_park 프로필** — 완료 거래 후기로 평점 표시
7. **admin 로그인** — 관리자 콘솔 → 재무 감사에서 원장·잔액 무결성 확인, 신고/분쟁 관리
8. **충전 시연** — 지갑 → 충전하기 → 금액 입력 → 결제창에서 승인 → 잔액 반영·원장 기록 확인

## 5. 유지보수

### 5.1 반복 개발로 확인·개선한 사항

**v1 사용 평가에서 발견한 문제 → v2에서 개선**

| 발견한 문제 | 단계 | 개선 |
|---|---|---|
| 직접 송금 구매는 사기(먹튀)에 무방비 | 요구사항 오류 | 에스크로 안전거래로 요구사항 변경·재설계 |
| 정찰제만 지원 → 실제 중고거래 관행(네고)과 불일치 | 요구사항 누락 | 가격 제안 기능 추가 |
| 판매자 신뢰도를 알 수 없음 | 요구사항 누락 | 거래 후기·평점 시스템 추가 |
| 상품이 늘면 목록만으로 탐색 불가 | 설계 한계 | 카테고리·필터·정렬·찜 추가 |
| 거래 이벤트를 당사자가 알 수 없음 | 설계 한계 | 알림함 추가 |
| Flask-SocketIO가 운영 모드 개발 서버 차단 | 구현 이슈 | 실습용 플래그 + 운영 배포 가이드 문서화 |
| 채팅이 새로고침 시 소실 | 구현 이슈 | 채팅 내역 DB 저장·로드 |
| 신고 시 UUID 직접 입력의 불편 | 사용성 | 대상이 미리 채워진 신고 링크 제공 |

**v2 테스트에서 발견한 문제 → 즉시 수정**

- 일일 한도 도입으로 기존 "잔액 초과" 테스트가 한도 검사에 먼저 걸림 → 테스트를 한도/잔액 2단계로 분리
- IP rate limit이 자동화 테스트(다수 로그인)와 충돌 → `IP_LOGIN_LIMIT` 환경변수로 조정 가능하게 개선

**v3 금융 강화 과정에서 발견한 문제 → 즉시 수정**

- 기존 에스크로 정산이 `상태 확인 → 잔액 지급` 2단계여서, 동시성 테스트(confirm 20회 동시 요청)로
  **이중 지급 가능성(TOCTOU)**을 실제로 재현 → 상태 조건부 단일 UPDATE로 재작성 (V31)
- 폼 토큰 도입 후 "예약중 상품 재구매" 테스트가 실패 → 분석 결과 예약중 상품은 구매 폼(토큰) 자체가
  렌더링되지 않아 재구매가 원천 불가한 **정상적 이중 방어**임을 확인, 테스트를 두 경로 모두 허용하도록 수정

**시연·요구사항 재점검에서 발견한 문제 → 수정**

- 요구사항 "등록된 상품은 누구나 볼 수 있어야 함" 재점검 결과, 상품 목록·상세·검색이 로그인 필수로
  구현되어 있었음(요구사항 미충족) → **비로그인 공개 열람으로 전환**하고, 구매·채팅·찜·판매 등
  상태 변경 액션은 로그인 필수로 유지. E2E에 미인증 공개열람/구매차단 케이스 추가
- 로컬에서 "사고 파는" 시연이 번거로움 → `seed.py`로 데모 계정·상품(중고 시세 반영)·사진·잔액을
  일괄 등록해 즉시 시연 가능하게 개선 (오프라인 시 플레이스홀더 폴백으로 재현성 확보)

**문서 검증(실제 실행)에서 발견한 문제 → 즉시 수정**

- 문서에 적은 배포 명령 `gunicorn -k eventlet`이 최신 gunicorn(23+)에서 **eventlet 워커 제거**로
  실패(`Entry point ('gunicorn.workers','eventlet') not found`) → **gevent + gevent-websocket**으로
  교체하고 `Dockerfile`·`Procfile`·`render.yaml`·문서를 모두 수정, Docker 빌드·구동까지 재검증
- `pkill -f "python app.py"`가 같은 문자열을 포함한 셸까지 종료시키는 footgun 확인 →
  `pkill -f "[p]ython app.py"` 형태로 문서 수정
