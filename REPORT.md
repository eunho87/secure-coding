# Tiny Second-hand Shopping Platform 개발 보고서

시큐어 코딩 과제 — 중고거래 플랫폼 개발 전 과정 보고서

- GitHub Repository: https://github.com/eunho87/secure-coding
- 개발 환경: WSL2 Ubuntu, Python 3.12, Flask 3.x
- 기반 코드: https://github.com/ugonfor/secure-coding (스켈레톤을 clone 후 확장)
- 개발 방식: 2회 반복(iteration) 개발 — v1(최소 요구사항+보안) → 사용성 평가 → v2(실거래 수준 기능 확장)

---

## 1. 요구사항 분석

### 1.1 기능 요구사항

과제의 최소 요구사항을 기준으로, 실제 중고거래 서비스(당근마켓·번개장터 등)를 벤치마킹하여
세부 요구사항을 도출했다.

| # | 최소 요구사항 | 도출한 세부 요구사항 (v1 → v2 확장) |
|---|---|---|
| R1 | 사람들이 플랫폼에 가입할 수 있어야 함 | 회원가입/로그인/로그아웃, 아이디 중복 방지, 마이페이지(소개글·비밀번호 변경), 사용자 프로필 조회, **2단계 인증(TOTP)** |
| R2 | 상품들을 올리고 볼 수 있어야 함 | 상품 등록(상품명·설명·가격·**카테고리·상품상태·사진 5장**), 목록/상세 페이지, 내 상품 관리, **조회수·찜하기**, 판매중/예약중/판매완료 상태 |
| R3 | 사용자들끼리 소통이 가능해야 함 | 실시간 전체 채팅, 1:1 채팅(내역 보존), **알림함** |
| R4 | 악성 유저나 상품을 차단해야 함 | 사용자/상품 신고(사유 필수), 신고 누적 자동 차단(상품 3회)/휴면(유저 5회), 관리자 수동 처리 |
| R5 | 유저들 간의 송금이 가능해야 함 | 지갑/충전/송금 + **에스크로 안전거래**(대금 보관→구매확정 시 지급, 취소/분쟁 중재), **가격 제안(네고)**, **거래 후기/평점**, 일일 거래 한도 |
| R6 | 상품 검색이 가능해야 함 | 키워드(제목+설명) + **카테고리/가격대 필터, 정렬(최신·저가·고가·인기)** |
| R7 | 관리자가 모든 요소를 관리할 수 있어야 함 | 사용자(휴면)/상품(차단·삭제)/신고 처리, **에스크로 분쟁 중재(지급/환불)**, 감사 로그 |

### 1.2 비기능 요구사항

- **보안**: 과제의 핵심. OWASP Top 10 계열 취약점이 없도록 설계 단계부터 반영 (3장, 5장)
- **신뢰성**: 결제·송금은 원자적으로 처리되어 이중 지불·잔액 음수·거래 중 상품 삭제가 불가능해야 함
- **사용성**: 카드형 상품 그리드, 카테고리 칩, 상태 뱃지 등 실서비스 수준의 화면 구성

### 1.3 요구사항 변경 이력 (유지보수 반영)

v1 사용 평가에서 "직접 송금 기반 구매는 사기에 취약하다(먹튀 가능)"는 문제를 확인
→ **에스크로 방식으로 요구사항을 변경**하고 v2에서 구현했다. (과제 조건: 논리적으로 타당한 요구사항 변경 가능)

## 2. 시스템 설계

### 2.1 아키텍처

```
[브라우저] ──HTTP──> [Flask 웹 서버 (app.py)] ──파라미터 바인딩──> [SQLite (market.db)]
     └──WebSocket(Socket.IO)──> [Flask-SocketIO (채팅)] ────────────────┘
```

### 2.2 웹페이지 설계

| 페이지 | 경로 | 설명 |
|---|---|---|
| 기본/가입/로그인 | `/`, `/register`, `/login`, `/login/2fa` | 인증 (2FA 단계 분리) |
| 홈(대시보드) | `/dashboard` | 카테고리 칩 + 상품 카드 그리드 + 전체 채팅 |
| 검색 | `/search` | 키워드/카테고리/가격대/정렬 필터 |
| 상품 | `/product/new`, `/product/<id>`, `/product/<id>/edit`, `/product/<id>/delete` | 등록(다중 사진)/상세(갤러리·찜·네고·안전거래)/수정/삭제 |
| 내 활동 | `/my/products`, `/my/favorites` | 내 상품 관리, 찜 목록 |
| 네고 | `/product/<id>/offer`, `/offer/<id>/respond` | 가격 제안/응답 |
| 지갑 | `/wallet`, `/wallet/charge`, `/wallet/transfer` | 잔액·충전·송금·거래내역·에스크로 현황 |
| 에스크로 | `/product/<id>/buy`, `/escrow/<id>/confirm·cancel·approve_cancel·dispute·review` | 결제→확정/취소/분쟁/후기 |
| 소통 | `/chats`, `/chat/<username>`, `/notifications` | 1:1 채팅, 알림함 |
| 마이페이지 | `/profile`, `/profile/password`, `/profile/2fa/*` | 소개글·비밀번호·2FA 관리 |
| 사용자 조회 | `/user/<username>` | 프로필·평점·후기·판매상품 |
| 신고 | `/report` | 사용자/상품 신고 |
| 관리자 | `/admin`, `/admin/users·products·reports·escrows·logs` | 전체 관리 + 분쟁 중재 |

### 2.3 데이터베이스 설계 (11개 테이블)

```
user          (id PK, username UNIQUE, password_hash, bio, balance,
               is_admin, is_dormant, failed_logins, locked_until,
               totp_secret, session_ver, must_change_pw, created_at)
product       (id PK, title, description, price INT, category, condition,
               view_count, status[selling/reserved/sold] CHECK, is_blocked,
               seller_id FK, created_at)
product_image (id PK, product_id FK ON DELETE CASCADE, filename, sort)
favorite      (user_id FK, product_id FK, PK(user_id, product_id))
offer         (id PK, product_id FK, buyer_id FK, amount CHECK(>0),
               status[pending/accepted/rejected] CHECK, created_at)
escrow        (id PK, product_id FK, buyer_id FK, seller_id FK, amount CHECK(>0),
               status[held/released/refunded/cancel_requested/disputed] CHECK,
               created_at, updated_at)
review        (id PK, escrow_id FK, reviewer_id FK, target_id FK,
               rating CHECK(1~5), comment, UNIQUE(escrow_id, reviewer_id))
notification  (id PK, user_id FK, content, link, is_read, created_at)
report        (id PK, reporter_id FK, target_type CHECK, target_id, reason,
               status, UNIQUE(reporter_id, target_type, target_id))
message       (id PK, room, sender_id FK, content, created_at)
transfer      (id PK, sender_id FK, receiver_id FK, amount CHECK(>0),
               memo, kind[transfer/purchase], created_at)
audit_log     (id PK, user_id, action, detail, ip, created_at)
```

설계 시 보안 고려사항:

- 비밀번호는 해시 컬럼으로만 저장, 금액은 INTEGER + CHECK 제약
- 상태 값은 전부 CHECK 제약(화이트리스트)으로 DB 수준에서 강제
- 신고·후기는 UNIQUE 제약으로 중복(남용) 방지
- PK는 추측 불가능한 UUIDv4 (순차 ID 열거 공격 방지)
- 1:1 채팅방 이름은 서버가 두 사용자 id를 정렬해 생성 (임의 방 접근 차단)

### 2.4 에스크로 상태 머신 설계

```
        구매(결제: 구매자 잔액 차감, 상품 reserved)
                        │
                      [held] ──구매자 취소 요청──> [cancel_requested]
                     /      \                        /        \
        구매자 확정 /        \ 분쟁                판매자 승인   \ 분쟁
                  ▼           ▼                      ▼           ▼
             [released]   [disputed] ──관리자 중재──> released 또는 refunded
          (판매자 지급,   (관리자만 처리 가능)      [refunded] (구매자 환불,
           상품 sold)                                상품 selling 복귀)
```

- 모든 상태 전이는 **행위자 권한 검증**(구매자만 확정, 판매자만 취소 승인, 관리자만 중재) 후 수행
- 거래 진행 중(reserved) 상품은 수정/삭제/중복 구매 불가

## 3. 시스템 구현

### 3.1 구현 스택

| 구성 | 선택 | 이유 |
|---|---|---|
| 웹 프레임워크 | Flask | 스켈레톤 유지, 경량 |
| 실시간 통신 | Flask-SocketIO | 전체/1:1 채팅 |
| DB | SQLite + 파라미터 바인딩 | 모든 쿼리 `?` 바인딩으로 인젝션 원천 차단 |
| CSRF | Flask-WTF (CSRFProtect) | 전역 CSRF 보호 |
| 비밀번호 해시 | bcrypt | 적응형 해시, salt 자동 |
| 2FA | 표준 라이브러리로 RFC 6238 TOTP 직접 구현 | 의존성 최소화, hmac.compare_digest로 타이밍 공격 방지 |

### 3.2 기능별 핵심 구현 포인트

- **인증**: 로그인 5회 실패 → 5분 잠금 + IP당 5분 10회 rate limit. 2FA 활성 계정은
  비밀번호 검증 후에도 세션을 발급하지 않고 별도 2FA 단계(시도 5회 제한)를 통과해야 로그인 완료
- **세션 버전**: `user.session_ver`를 세션에 함께 저장하고 매 요청 비교.
  비밀번호 변경 시 버전 증가 → 탈취된 세션 포함 다른 기기 세션 전부 즉시 무효화
- **에스크로**: 결제 시 `UPDATE ... SET balance = balance - ? WHERE id = ? AND balance >= ?`
  단일 문으로 잔액 검사+차감을 원자화(이중 지불 방지). 대금은 판매자에게 바로 가지 않고
  escrow 레코드로 보관되며, 정산(지급/환불)은 상태 머신 검증을 통과한 요청만 수행
- **가격 제안**: 판매가 미만 검증, 구매자당 1건의 pending 제안, 판매자만 수락/거절 가능(403).
  수락된 제안가는 구매 시 서버가 다시 조회해 적용 (클라이언트가 가격을 보내지 않음)
- **후기**: 거래 완료(released) 상태에서만, 거래 당사자만, 거래당 1회(UNIQUE 제약)
- **파일 업로드**: 확장자 화이트리스트 + 매직바이트 검증 + 랜덤 파일명 + 장수(5)/용량(10MB) 제한,
  업로드 파일 서빙 경로는 파일명 형식 검증(정규식) 후 인증 사용자에게만 제공
- **알림**: 서버가 생성한 내용/링크만 저장 (사용자 입력 미포함)

### 3.3 발견한 보안 약점과 수정 내역

#### 스켈레톤 코드에서 발견한 약점 (v1에서 수정)

| # | 보안 약점 (스켈레톤) | 위험 | 수정 내용 |
|---|---|---|---|
| V1 | 비밀번호 평문 저장·평문 비교 (`WHERE username=? AND password=?`) | DB 유출 시 전체 계정 탈취 | bcrypt 해시+salt 저장, `bcrypt.checkpw` 비교 |
| V2 | `SECRET_KEY = 'secret!'` 하드코딩 | 세션 쿠키 위조 → 임의 계정 탈취 | 환경변수 또는 랜덤 32바이트 키(`instance/secret_key`, 0600) |
| V3 | CSRF 보호 전무 | 로그인 사용자를 속여 임의 요청(송금 등) 실행 | Flask-WTF `CSRFProtect` 전역 적용 |
| V4 | 서버측 입력 검증 없음 | XSS·데이터 오염 | 정규식·길이·범위·화이트리스트 검증 일괄 적용 |
| V5 | 채팅에서 클라이언트가 보낸 `username` 신뢰 | 사용자 사칭 | 서버 세션에서 사용자명 조회 |
| V6 | Socket.IO 인증 없음 | 비로그인 채팅 참여 | `connect` 시 세션 검증 + 이벤트마다 재검증 |
| V7 | 채팅 rate limit 없음 | 스팸/도배 | 사용자당 10초 5건 제한 |
| V8 | `debug=True` 운영 실행 | 스택트레이스·디버거 노출(RCE 가능) | `debug=False` + 커스텀 에러 페이지 |
| V9 | 세션 쿠키 보호 플래그·만료 없음 | 세션 탈취 | HttpOnly, SameSite=Lax, Secure(HTTPS), 30분 만료 |
| V10 | 로그인 실패 제한 없음 | 무차별 대입 | 5회 실패 시 5분 계정 잠금 |
| V11 | 로그아웃이 GET 요청 | CSRF 강제 로그아웃 | POST + CSRF 토큰 |
| V12 | `price`가 TEXT 컬럼 | 음수·문자열 가격 | INTEGER + CHECK + 서버 검증 |
| V13 | 소유자 확인 없는 리소스 접근 | IDOR | 소유자/당사자 검증, 실패 시 403 |
| V14 | 신고 남용 방지 없음 | 신고 폭탄 | UNIQUE 제약, 자기/관리자 신고 금지, 신고자 수 기준 집계 |
| V15 | 보안 헤더 없음 | 클릭재킹, 스크립트 주입 | CSP(nonce), X-Frame-Options, nosniff, Referrer-Policy |
| V16 | 세션 고정 대응 없음 | 세션 탈취 | 로그인 성공 시 `session.clear()` 후 재발급 |
| V17 | LIKE 와일드카드 미이스케이프 | LIKE 인젝션 | `%`, `_`, `\` 이스케이프 후 바인딩 |
| V18 | 파일 업로드 검증 없음 | 웹셸 업로드 | 화이트리스트+매직바이트+랜덤 파일명+용량 제한 |
| V19 | 송금 동시성 문제 | 이중 지불, 잔액 음수 | 조건부 원자적 UPDATE, CHECK 제약 |
| V20 | 감사 추적 불가 | 침해사고 대응 불가 | 감사 로그(audit_log) 기록 |

#### v2 확장 과정에서 추가로 식별·보완한 약점

| # | 식별한 약점/위협 | 위험 | 보완 내용 |
|---|---|---|---|
| V21 | 직접 송금 기반 구매 → 판매자 미발송(먹튀) 시 구제 불가 | 거래 사기 | **에스크로 도입**: 대금 플랫폼 보관, 구매 확정 시 지급, 분쟁 시 관리자 중재 |
| V22 | 비밀번호 유출 시 단일 인증 요소로 계정 탈취 | 계정 탈취 | **TOTP 2단계 인증** (선택 활성화, 시도 횟수 제한, 타이밍 공격 방지 비교) |
| V23 | 비밀번호 변경 후에도 탈취된 세션이 유효 | 세션 탈취 지속 | **세션 버전** 도입 — 변경 시 전 기기 세션 즉시 무효화 |
| V24 | 계정 탈취 시 잔액 전액 탈취 가능 | 금전 피해 확대 | **일일 거래 한도**(500만원)로 피해 상한 제한 |
| V25 | 계정 잠금이 계정 단위뿐 → 다계정 대입 공격 가능 | 무차별 대입 | **IP 단위 로그인 rate limit** 추가 |
| V26 | 관리자 기본 비밀번호 방치 가능성 | 관리자 계정 탈취 | 기본 비밀번호로 생성 시 **최초 로그인에서 변경 강제** |
| V27 | 에스크로/제안 상태 전이를 임의 사용자가 수행할 가능성 | 권한 상승·자금 탈취 | 상태 머신 + **행위자 검증**(구매자/판매자/관리자 역할별 403) 및 E2E 테스트로 확인 |
| V28 | 수락된 제안가를 클라이언트가 위조할 가능성 | 가격 조작 | 결제 금액은 **서버가 DB에서 재조회** (클라이언트 입력 미사용) |
| V29 | HTTPS 환경에서 HSTS 부재 | SSL 스트리핑 | `SESSION_COOKIE_SECURE=1` 시 HSTS 헤더 자동 적용 |
| V30 | 정렬/카테고리 파라미터를 쿼리에 직접 사용 시 인젝션 | SQL 인젝션 | 정렬키·카테고리 **화이트리스트 매핑** 후 사용 (임의 값 400) |

#### v3 금융 거래 보안 강화 과정에서 식별·보완한 약점

실제 자산이 오가는 거래 시스템으로 발전시키면서, 금융 도메인 특유의 위협을 집중적으로 보완했다.

| # | 식별한 약점/위협 | 위험 | 보완 내용 |
|---|---|---|---|
| V31 | 에스크로 정산이 `상태 확인 → UPDATE` 2단계라 동시 요청 시 **이중 지급** 가능 (TOCTOU race condition) | 자금 이중 인출 | 정산을 **상태 조건부 단일 UPDATE**(`WHERE status IN(...)`)로 원자화. `rowcount==1`일 때만 지급/환불. 동시 confirm 20회 요청에도 정확히 1회만 정산됨을 테스트로 검증 |
| V32 | 잔액 변동이 `user.balance`에만 반영되어 **감사·정합성 검증 불가** | 내부 부정·오류 탐지 불가 | 모든 잔액 변동을 거치는 단일 함수 `apply_balance()` 도입, **이중기입 원장(ledger)**에 변동액과 변동 후 잔액 기록. 관리자 재무 감사 페이지에서 원장 합계 vs 실제 잔액 대조 |
| V33 | 송금/결제 폼 **더블클릭·새로고침 재전송** 시 중복 거래 | 의도치 않은 중복 이체 | 거래 폼마다 **1회용 서버 토큰**(form_token) 발급, 소비 후 재사용 차단 (E2E로 재사용 차단 확인) |
| V34 | 비밀번호만으로 송금·결제 승인 → 비밀번호 유출 시 즉시 자금 이동 | 자금 탈취 | 2FA 활성 사용자는 **거래 시 TOTP 코드 추가 요구**(송금·구매 재인증에 통합) |
| V35 | 탈취 계정으로 **단시간 다수 출금**(velocity) 및 고액 이체 무제한 | 대량 자금 유출 | 간단한 **이상거래탐지(FDS)**: 10분 내 5회 초과 출금 차단, 고액(100만원 이상) 거래는 관리자에게 자동 알림 |
| V36 | DB 레벨에서 음수 잔액을 막는 최종 방어선 부재 | 데이터 무결성 훼손 | `user.balance`에 **`CHECK(balance >= 0)` 제약** 추가 (애플리케이션 검증이 뚫려도 DB가 최종 차단) |
| V37 | 결제 금액을 애플리케이션 로직에만 의존 | 잔액 초과 결제 | 잔액 차감을 `balance >= ?` 조건부 UPDATE로만 수행해 초과 인출 원천 차단 (V19 원칙을 전 거래 경로에 일관 적용) |

**금융 보안 설계 원칙 요약**: ① 모든 잔액 변동은 `apply_balance()` 단일 통로만 통과 ② 잔액 검사와 차감은 **한 SQL 문 안에서** 원자적으로 ③ 상태 전이는 조건부 UPDATE로 멱등하게 ④ 모든 변동은 원장에 남겨 사후 검증 가능 ⑤ 애플리케이션·DB 이중 방어.

#### v4 실제 PG(결제) 연동 과정에서 식별·보완한 약점

지갑 충전을 "버튼 누르면 즉시 잔액 증가"에서 **실제 PG 결제 흐름**(주문 생성 → 결제창 →
승인 콜백)으로 바꾸면서, 결제 도메인의 대표적 공격을 방어했다.

| # | 식별한 약점/위협 | 위험 | 보완 내용 |
|---|---|---|---|
| V38 | 승인 콜백의 **금액을 클라이언트가 위·변조**(100원 결제 후 100만원 크레딧 요청) | 부정 충전 | 승인 시 콜백 금액이 **서버에 저장된 주문 금액과 정확히 일치**할 때만 처리 (불일치 시 거부) |
| V39 | 결제 승인 콜백 **중복 수신/새로고침**으로 이중 충전 | 부정 충전 | `pending→paid` 조건부 UPDATE로 **멱등 처리** (동시 20회 승인 요청에도 1회만 크레딧 — 테스트 검증) |

| V41 | PG 승인을 클라이언트 응답만 믿고 처리 | 위조된 성공 응답 | 토스 결제는 **시크릿 키로 서버-서버 확인 API** 호출로 최종 승인 (시크릿은 클라이언트에 절대 노출 안 함) |
| V42 | 결제창(외부 스크립트) 도입으로 CSP 무력화 우려 | XSS 방어 약화 | 토스 도메인은 `PAYMENT_PROVIDER=toss`일 때만 CSP에 **최소 범위로 허용**, 기본(mock)은 strict 유지 |
| V43 | 결제 주문 폼 **더블클릭**으로 중복 주문 | 중복 결제 | 주문 생성에도 1회용 폼 토큰 적용 (V33 확장) |

### 3.4 잔여 위험 및 한계 (정직한 평가)

- 개발용 Werkzeug 서버로 구동됨. 운영에는 gunicorn+gevent-websocket 및 HTTPS(TLS) 종단 필요
  → `Dockerfile`/`Procfile`/`render.yaml`로 제공, [실행방법.md](실행방법.md) 5장 명시
- rate limit이 프로세스 메모리 기반 → 다중 프로세스 배포 시 Redis 등으로 교체 필요
- 잔액/충전은 모의 재화. 실 결제 연동 시 PG사 API·본인확인(KYC)·정식 이상거래탐지(FDS) 고도화가 추가로 필요 (현재 FDS는 velocity/고액 알림 수준의 기본형)
- 2FA 복구 코드(백업 코드) 미구현 → 인증 앱 분실 시 관리자 개입 필요
- 원장·잔액이 단일 SQLite 트랜잭션으로 정합성을 유지하나, 다중 프로세스/분산 배포 시에는 트랜잭션 격리 수준·분산 락 검토 필요

### 4.2 테스트 케이스 명세 및 자동화

요구사항·보안·금융별 테스트 케이스 표는 [TESTCASES.md](TESTCASES.md)에 정리했고,
아래 3개 스크립트로 실제 HTTP 요청·DB 조회로 자동 검증한다.
`tests/e2e_test.sh`(67개) + `tests/extra_test.sh`(8개) + `tests/finance_test.sh`(6개).

- **기능**: 가입→로그인→상품 등록(카테고리/사진)→**비로그인 공개 열람**→검색(필터/정렬)→찜→

  (요청→승인→환불)→신고→관리자 처리→알림 수신→**PG 결제 충전** 전체 시나리오
- **일반 보안**: CSRF 차단, SQL 인젝션(로그인/검색), XSS 이스케이프, IDOR(상품/제안/에스크로/결제 403·404),

  로그인 불가→코드 검증), 관리자 초기 비밀번호 변경 강제, 에러 페이지 정보 비노출
- **금융 보안**: 1회용 폼 토큰 이중 제출 차단, 일일 한도, 잔액 초과 차단, 원장 표시,

  원장-잔액 정합성, 음수 잔액 불가(CHECK 제약)

**v2 테스트에서 발견한 문제 → 즉시 수정:**

- 일일 한도 도입으로 기존 "잔액 초과" 테스트가 한도 검사에 먼저 걸림 → 테스트 케이스를 한도/잔액 2단계로 분리
- IP rate limit이 자동화 테스트(다수 로그인)와 충돌 → `IP_LOGIN_LIMIT` 환경변수로 조정 가능하게 개선

**v3 금융 강화 과정에서 발견한 문제 → 즉시 수정:**

- 기존 에스크로 정산이 `상태 확인 → 잔액 지급` 2단계여서, 동시성 테스트(confirm 20회 동시 요청)로 **이중 지급 가능성(TOCTOU)**을 실제로 재현 → 상태 조건부 단일 UPDATE로 재작성해 멱등 정산으로 수정 (V31)
- 폼 토큰 도입 후 "예약중 상품 재구매" 테스트가 실패 → 분석 결과 예약중 상품은 구매 폼(토큰) 자체가 렌더링되지 않아 재구매가 원천 불가한 **정상적 이중 방어**임을 확인, 테스트를 두 방어 경로 모두 허용하도록 수정

**시연·요구사항 재점검에서 발견한 문제 → 수정:**

- 요구사항 "등록된 상품은 누구나 볼 수 있어야 함"을 재점검한 결과, 상품 목록·상세·검색이 로그인 필수로 구현되어 있었음(요구사항 미충족) → **비로그인 공개 열람으로 전환**하고, 구매·채팅·찜·판매 등 상태 변경 액션은 로그인 필수로 유지. 익명 요청이 액션 엔드포인트에 닿지 못하도록 뷰·템플릿을 이중 점검(E2E에 미인증 공개열람/구매차단 케이스 추가)
- 로컬에서 "사고 파는" 시연이 번거로움 → `seed.py`로 데모 계정·상품(중고 시세 반영)·사진·잔액을 일괄 등록해 즉시 시연 가능하게 개선. 상품 사진은 카테고리 매칭 무료 이미지를 받고, 오프라인 시 플레이스홀더로 폴백해 재현성 확보
